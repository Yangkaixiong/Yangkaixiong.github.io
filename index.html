1.需求分析

这一部分主要是对为什么要进行个人博客部署进行分析，从功能需求和性能需求两方面来进行。

1.1 功能需求

  • 避免重复性劳动：在学习和工作过程中，我们经常遇到一些这样或那样的问题，此时我们可能会在网上找到相应的解决方法。但是过了一段时间之后，当我们再次碰到类似的问题时，早已忘记以前是怎么解决的了，于是又要到网上去搜，浪费大量的时间和精力。
  • 随时随地可查。
  • 归纳总结：通过记录解决问题的过程，对问题进行复盘。
  • 进一步加深学习的程度：学习的过程就是先学习再实践，再学习，再实践的过程，而向别人讲清楚你所学的知识就是实践方法之一。

1.2 性能需求

  • 方便撰写和发布：既可以在手机上，也可以在电脑上撰写。
  • 安全性好。

2.现状分析

目前，主流的博客主要分静态博客和动态博客两类：所谓动态，就是指有前端有后端，可以登录到后台进行管理。静态博客是纯前端的展示，没有后端。

2.1 动态博客

2.1.1 CSDN、知乎等公共平台
  • 优点：会打字就行、阅读量高。
  • 缺点：限制多。
2.1.2 WordPress等框架
  • 优点：限制少、定制感强。
  • 缺点：笨重、有点门槛。
2.1.3 使用Python等自己开发
  • 优点：功能强大、获得极大成就感。
  • 缺点：门槛高。

2.2 静态博客

2.2.1 使用hexo、hugo、jekyll等框架 + github page
  • 优点：易上手、免费。
  • 缺点：访问慢。
2.2.2 使用hexo、hugo、jekyll等框架 +云服务器
  • 优点：访问速度快。
  • 缺点：要花点money。

2.3 总结

经过以上的分析，本文选用两种方式部署，一种是hexo + 腾讯云服务器，一种是hexo + github page。

二.指令

1.概念：

指令是==为了方便控制cpu而创造的==机器码。后来为了可读性，有了汇编语言和高级语言。

从高级语言到机器码要经过编译和汇编的过程如下：

  • 编译：将源文件变成汇编文件。

  • 汇编：将汇编文件变成二进制文件。

  • 链接：将多个二进制文件和函数库变成可执行文件。

    • 可执行文件分为ELF格式和PE格式，ELF格式文件可分为代码段，数据段，重定位表，符号表。
    • ==链接的具体过程为==：链接器先将所有ELF文件中的符号表信息汇总，然后通过符号表将重定位表中未确定的跳转地址进行修正，最后把所有文件的对应段合并并生成可执行文件。

  • 加载：将ELF文件解析并把指令和数据加载进内存。（==将指令和数据加载进内存要解决两个问题==，一是内存必须连续，二是内存地址和程序想要的地址必须对应）

    • 首先，我们采用内存分段 + 内存映射的办法解决。在物理内存空间中找到一段连续的内存空间并把这段内存的地址（物理地址）和程序指定的地址（虚拟地址）映射起来。但是这样做有一个问题就是，产生内存碎片。
      • 注意，内存映射不是一个字节一个字节映射，而是一头一尾的映射。
    • 然后，我们采用内存分段 + 内存映射 + 内存交换的办法解决。通过把，但是这种方法会造成卡顿。
    • 最后，我们采用内存分页 + 内存映射 + 内存交换的办法解决。把整个物理空间划分成固定大小且连续的页，然后进行映射，当内存空间不够用的时候，只需要交换少数的页，因此不会造成卡顿。除此之外，内存分页带来的好处是，在物理地址和虚拟地址映射完后无需加载真的指令和数据，可以等到真正需要的时候再加载。

    四.函数的调用

1.第一种实现函数调用的方法（内联）：

将那些不会发生无穷无尽情况的函数生成的指令==直接替换掉==该函数的调用指令。这样做的好处是降低栈的开销，坏处是增加了程序占用的内存大小。
  • 可以让编译器主动内联，也可以人为指示内联。

    2.第二种实现函数调用的方法（程序栈）：

    我们可以将函数被调用后要返回的地址==单独记录==下来以此来实现函数调用，但是可能会发生栈溢出，因此我们可以==使用内存中的栈==，在调用前将调用返回后的地址入栈，在调用完后将出栈并跳转到调用返回后的地址。
